## 1. 产品概述

### 1.1 产品愿景

打造一个**纯本地运行**的 CLI 工具，将任意本地代码仓库转化为**交互式"学习百科"**——不只是自动生成结构化文档和图表，更要像一位资深导师那样，帮助开发者**理解**代码库的设计理念、架构决策和使用方式。代码永不离开本机。

> **产品哲学转变**：从"代码结构的说明书"转向"帮你学会这个代码库的老师"。我们生成的不是模块清单和函数列表，而是围绕"概念→决策→实现"的叙事式学习体验。

### 1.2 目标用户

| 用户群体 | 核心痛点 | 使用场景 |
|----------|----------|----------|
| 新加入团队的开发者 | 面对庞大代码库无从下手，缺乏全局认知 | 本地运行，快速理解项目架构与核心模块 |
| 资深开发者/架构师 | 接手遗留代码、需要快速理清模块关系 | 自动生成架构图和调用链，可视化理解代码 |
| 安全敏感场景的开发者 | 代码不能上传到第三方平台 | 纯本地运行 + 本地 LLM 部署，代码零泄露 |
| 开源项目维护者 | 需要让贡献者更快上手项目 | 生成 Wiki 提交到仓库，作为项目入口文档 |

### 1.3 核心价值主张

- **完全本地**：代码分析、文档生成、问答全在本地完成，代码不出本机
- **双模 LLM**：支持 API 调用（OpenAI/Claude/Gemini 等）或本地部署模型（Ollama/LocalAI）
- **学习体验优先**：生成的文档按"概念→架构→实现"的叙事组织，附带学习路径引导，而非模块清单罗列
- **图表可视**：自动生成架构图、类图、时序图，不只是文字描述
- **设计决策显性化**：主动提取并呈现关键架构决策、设计哲学和权衡取舍
- **一键生成**：单条命令完成从分析到 Wiki 的全流程
- **自然交互**：内置问答终端，用自然语言探索代码库

## 2. 竞品深度分析

### 2.1 Google Code Wiki (codewiki.google)

**产品定位**：由 Google 推出的云端自动化代码文档平台，前身为 MutableAI 的 Auto Wiki（被 Google 收购）。

**核心能力**：
- **三大支柱**：自动更新 / 上下文感知 / 可操作性
- **自动图表生成**：架构图、类图、时序图（唯一确认支持三种图表的竞品）
- **Gemini 驱动**：使用 Gemini 模型作为 AI 后端
- **NotebookLM 集成**：可将代码 Wiki 转为播客/音频格式
- **超链接溯源**：每个 Wiki 段落直接链接到源代码文件和行号
- **Gemini CLI 扩展**（开发中）：支持私有仓库的本地分析

**技术栈**：
- 前端：Angular SPA (`BoqAngularSdlcAgentsUi`)
- AI 引擎：Google Gemini
- GitHub 集成：需授权仓库读取权限并上传至 Google 服务器

**局限性**（来自 HN 社区反馈，101 points / 41 comments）：
- 仅支持公开仓库（私有仓库待 CLI 扩展）
- 代码需上传至 Google 服务器，安全敏感场景不可用
- 单页面布局，"信息墙"缺乏视觉吸引力
- 仓库需逐个请求/索引，非自动化
- 文档变化不可预测（重复生成结果可能不同）
- 谷歌产品关停历史引发的信任担忧

### 2.2 Zread (zread.ai) —— 学习体验标杆

**产品定位**：由智谱 AI (CodeGeeX) 推出的 AI 代码 Wiki CLI 工具，支持多语言和深度研究。

**核心能力**：
- **CLI 为本**：核心产品是命令行工具 `zread`，在本地项目目录运行即可生成文档
- **多模型支持**：智谱/OpenAI/MoonShot/MiniMax/OpenRouter/自定义
- **多语言输出**：支持中英文界面与文档生成
- **本地存储**：文档保存在 `.zread/wiki/`，不上传云端
- **续写机制**：中断后可从草稿继续，避免重复生成
- **版本历史**：每次生成保留历史版本

**技术栈**：
- CLI：Go 语言编写
- 安装方式：Homebrew / npm / winget / 直接下载
- 跨平台：macOS / Linux / Windows (x64/ARM64)
- 配置：YAML 格式 (`~/.zread/config.yaml`)

**Zread 的学习体验设计（我们的核心对标对象）**：

Zread 的 Wiki 页面体验代表了我们追求的"学习百科"方向：

1. **主题导向组织**：文档不按文件/模块罗列，而是按"概念/主题"组织——"What it Does"、"Architecture at a Glance"、"Output Philosophy"、"Authentication"
2. **叙事式内容**：每篇文章回答三个问题——"这是什么？为什么这样设计？对你意味着什么？"
3. **学习路径引导**：明确的"Where to Go Next"，按用户目标（快速上手/部署/理解内部）分支引导
4. **设计决策显性化**：有专门的"Key architectural decisions"段落，解释为什么选择当前架构
5. **来源溯源**：每个段落末尾标注 Sources，建立文档可信度
6. **表格总结**：用表格做能力对比、配置对比，而非纯文本罗列
7. **难度分级**：标注 Level（Beginner/Intermediate/Advanced）

**Zread 的 UI/UX 设计借鉴（我们的 Web UI 对标对象）**：

除了内容组织哲学，Zread 的 Web UI 也提供了重要的交互设计参考：

| 设计元素 | Zread 表现 | 我们借鉴的要点 |
|----------|-----------|---------------|
| **阅读时长预估** | 每篇文章顶部标注预计阅读时间（如"5 min read"） | 基于中文字数统计（400 字/分钟），每篇文章注入 ⏱ 阅读分钟数徽章 |
| **难度星级标注** | 文章标题旁标注 Level（Beginner / Intermediate / Advanced） | 根据文件名前缀分级（⭐入门 / ⭐⭐进阶 / ⭐⭐⭐高级），带颜色编码徽章 |
| **导航分层折叠** | 左侧导航按学习阶段分组，可折叠展开 | 导航分为"⭐ 入门""⚡ 动态""🔭 探索"三组，每组可独立折叠/展开 |
| **来源溯源** | 段落末尾标注 Sources，链接到源文件 | 叙事 Prompt 要求每段末尾标注 `*来源：[文件路径:行号]*` |
| **下一步引导** | "Where to Go Next" 明确给出下一篇文章 | 每篇 Wiki 文章底部注入"下一步阅读 →"导航 |
| **代码块可读性** | 代码块有语言标签，结构化展示 | 代码块顶部显示语言标签 + 一键复制按钮 |
| **主题切换** | 支持暗色模式 | 亮色/暗色双主题，使用 CSS 自定义属性 + localStorage 持久化 |
| **图表交互** | —（Zread 无图表） | 图表支持全屏展开、Mermaid 点击跳转到源码模块 |

> **关键启发**：Zread 证明了开发者愿意为"好的阅读体验"付费——不仅仅是内容质量，还包括排版、导航、视觉层次和交互细节。我们的策略是：将 Zread 的**内容组织哲学**（主题导向、叙事式、学习路径）与 Google Code Wiki 的**可视化能力**（架构图、类图、时序图）结合起来，创造出两者优势兼备的产品。

**局限性**：
- **不生成可视化图表**：仅有文本架构描述，不生成架构图/类图/时序图
- 不支持本地 LLM 部署，必须使用 API
- 社区活跃度低（HN 仅 3 分，零评论）

### 2.3 其他竞品

| 产品 | 特点 | 不足 |
|------|------|------|
| **DeepWiki** (deepwiki.com) | Cognition 出品，Devin 驱动 | 云端，细节有限 |
| **Doxygen** | 从源码注释生成文档 | 需规范注释，无 AI 问答，无图表 |
| **Sphinx** | Python 文档生成器 | 手动配置，无自动分析 |

### 2.4 竞品差距与我们的机会

```
                自动图表   纯本地运行   本地LLM   多语言输出   AI问答   学习体验
Google Code Wiki   ✅        ❌          ❌         ❌        ✅        ❌
Zread              ❌        ✅          ❌         ✅        ✅        ✅
CodeWiki (我们)    ✅        ✅          ✅         ✅        ✅        ✅
```

**核心差异化**：
1. **学习体验优先**：唯一同时支持"主题导向叙事文档 + 学习路径引导 + 设计决策显性化"的代码 Wiki 工具
2. **可视化优先**：唯一同时支持架构图/类图/时序图自动生成 + 纯本地运行的工具
3. **本地 LLM 双模**：既能接 API 也能用 Ollama 等本地模型，适应不同安全需求
4. **代码零泄露**：100% 本地运行，源码和文档均不离开本机
5. **国际化**：中英文双语文档生成
6. **开放模型**：不绑定特定 LLM 提供商，兼容 OpenAI 接口的任何模型

## 3. 核心功能

### 3.1 功能全景

```
CodeWiki CLI
├── 代码分析引擎
│   ├── AST 解析（多语言 tree-sitter）
│   ├── 依赖关系图构建
│   ├── 调用链追踪
│   └── 代码元数据索引
├── 文档生成引擎
│   ├── 项目概览
│   ├── 架构文档
│   ├── 模块/包文档
│   ├── API 参考
│   └── 多语言输出（中/英）
├── 图表生成引擎
│   ├── 架构图（系统拓扑/模块关系）
│   ├── 类图（UML 风格）
│   ├── 时序图（调用链/交互流程）
│   └── 依赖图（模块间 import 关系）
├── 交互式问答
│   ├── RAG 检索增强生成
│   ├── 代码上下文感知
│   ├── 源码溯源（答案→代码行）
│   └── 多轮对话会话
└── LLM 适配层
    ├── 远程 API 模式（OpenAI/Claude/Gemini/智谱等）
    ├── 本地部署模式（Ollama/LocalAI/llama.cpp）
    └── 自定义 BaseURL（兼容 OpenAI 接口）
```

### 3.2 功能详细说明

#### 3.2.1 代码分析引擎

| 功能 | 说明 | 优先级 |
|------|------|--------|
| 多语言 AST 解析 | 通过 tree-sitter 支持 Python/JS/TS/Go/Java/Rust/C++ 等主流语言 | P0 |
| 依赖关系图构建 | 分析模块/包之间的 import 依赖关系，构建全局依赖图 | P0 |
| 函数调用链追踪 | 跨文件追踪函数调用链，识别数据流向 | P1 |
| 元数据索引 | 提取全部函数签名、类型定义、类结构、注释等结构化信息 | P0 |

#### 3.2.2 文档生成引擎

| 功能 | 说明 | 优先级 |
|------|------|--------|
| 项目概览 | 自动生成项目定位、技术栈识别、目录结构说明 | P0 |
| **主题导向文档** | **按"概念/主题"组织文档（What it Does / Architecture / Key Concepts），而非按文件/模块罗列** | **P0** |
| **学习路径生成** | **自动生成"Where to Go Next"，按用户目标（快速上手/部署/理解内部）分支引导** | **P0** |
| **设计决策提取** | **主动提炼并呈现关键架构决策、设计哲学、权衡取舍（Why 而不仅是 What）** | **P0** |
| 架构文档 | 描述系统分层架构、核心模块职责、设计模式 | P0 |
| API 参考 | 按模块输出函数/类/接口的签名、参数、返回值 | P0 |
| 模块文档 | 每个包/模块的职责、依赖关系、关键实现说明 | P1 |
| 函数级逻辑描述 | 每个函数/方法的核心逻辑、输入输出语义、边界条件说明 | P1 |
| 调用链路语义 | 时序图/调用链附带场景描述（触发条件、数据变换、异常路径） | P1 |
| 数据流文档 | 描述数据在系统中的流转路径与变换逻辑 | P2 |
| 多语言输出 | 支持中/英文文档生成，可配置 | P1 |

#### 3.2.3 图表生成引擎 —— 核心差异化能力

| 图表类型 | 说明 | 生成方式 | 优先级 |
|----------|------|----------|--------|
| **架构图** | 系统拓扑、模块关系、层次结构 | 依赖图 → 社区检测分层 → LLM 辅助布局 → Mermaid DSL | P0 |
| **类图** | UML 风格，类/接口定义及关系 | AST 提取类结构 → 关系推断 → 自动布局 → Mermaid DSL | P0 |
| **时序图** | 关键交互流程、调用序列 | 调用链追踪 → 关键路径提取 → LLM 补充 → Mermaid DSL | P0 |
| **依赖图** | 模块间 import 依赖关系全貌 | AST 静态分析 → 直接渲染 | P1 |
| **数据流图** | 数据处理 pipeline | 静态分析 + LLM 推断 | P2 |

图表统一输出为 **Mermaid DSL**，原因：
- 纯文本格式，可存入 Git、支持 Diff
- Markdown 原生支持，GitHub/GitLab 直接渲染
- 交互式浏览（Mermaid.js 支持缩放/平移/点击）
- 生成后可导出 SVG/PNG

#### 3.2.4 交互式问答

| 功能 | 说明 | 优先级 |
|------|------|--------|
| 自然语言提问 | 在终端中用自然语言询问代码相关问题 | P0 |
| RAG 检索 | 基于代码向量索引 + 文档索引的检索增强生成 | P0 |
| 源码溯源 | 每个回答附带源文件路径和行号 | P0 |
| 多轮对话 | 支持连续追问，保持上下文 | P1 |
| 图结构查询 | 基于代码图谱的结构化查询（"哪些模块依赖了 X？"） | P2 |

#### 3.2.5 LLM 适配层

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| **远程 API** | OpenAI / Claude / Gemini / 智谱等兼容 OpenAI 接口的服务 | 日常开发，网络畅通 |
| **本地部署** | Ollama / LocalAI / llama.cpp，或任意 OpenAI 兼容本地服务 | 安全敏感场景，离线环境 |
| **自定义 BaseURL** | 填入任意兼容地址，支持代理/中转 | 企业内网、定制化部署 |

LLM 策略分层：
- **轻量任务**（函数摘要、代码分块）→ 本地小模型 / 廉价 API
- **中等任务**（架构描述、图表 DSL 生成）→ 中等模型
- **重型任务**（全局一致性审查、复杂模式识别）→ 强模型

### 4.2 核心技术选型

| 层次 | 技术选择 | 理由 |
|------|----------|------|
| **CLI 语言** | Go | 单二进制分发，零依赖，跨平台，高性能 |
| **AST 解析** | tree-sitter | 50+ 语言支持，增量解析，错误恢复 |
| **图表 DSL** | Mermaid | 纯文本、Markdown 原生渲染、交互式 |
| **本地向量库** | SQLite + sqlite-vec / Chroma 嵌入式 | 零配置，本地文件存储，无服务依赖 |
| **图结构存储** | 内存图 + JSON 持久化 | 依赖关系天然适合图结构，无需外部 DB |
| **文档格式** | Markdown + YAML frontmatter | Git 友好，可读可版本管理 |
| **本地 Web UI** | 内嵌静态资源 (Go embed) + Mermaid.js | serve 命令一键启动，零配置浏览器预览 |
| **LLM 客户端** | Go OpenAI 兼容客户端 + Ollama API | 统一接口，适配远程和本地模型 |
| **安装分发** | Go 编译单二进制 + Homebrew/npm/winget | 成熟模式 |

### 4.3 数据流

```
本地代码仓库
    │
    ▼
┌─────────────┐     ┌──────────────┐     ┌──────────────┐
│ AST 解析     │────▶│ 代码图谱构建   │────▶│ 图表数据提取  │
│ (tree-sitter)│     │ (内存图结构)   │     │ (Mermaid DSL) │
└─────────────┘     └──────────────┘     └──────┬───────┘
       │                     │                    │
       ▼                     ▼                    ▼
┌─────────────┐     ┌──────────────┐     ┌──────────────┐
│ 语义分块     │     │ 文档草稿生成  │     │ 图表渲染     │
│ (Code Chunk) │     │ (LLM 生成)   │     │ (Web UI 中)  │
└──────┬──────┘     └──────┬───────┘     └──────────────┘
       │                    │
       ▼                    ▼
┌─────────────┐     ┌──────────────┐
│ 向量索引     │     │ Wiki 文件     │
│ (本地 VDB)   │     │ (.md + JSON) │
└──────┬──────┘     └──────────────┘
       │
       ▼
┌─────────────┐
│ RAG 问答     │
│ (codewiki ask)│
└──────────────┘

所有数据仅在本地流转。远程 API 模式下仅发送结构化的 Prompt 摘要给 LLM，不发送完整源码。
本地 LLM 模式下，100% 离线运行，零网络请求。
```

### 4.4 关键设计决策

1. **Mermaid DSL 作为图表的"中间表示"**
   - 文本化：可存入 Git，支持 Diff
   - 跨平台：Markdown 原生支持，GitHub/GitLab 直接渲染
   - 交互性：Mermaid.js 支持缩放、平移

2. **AST 静态分析 + LLM 语义理解的混合策略**
   - 静态分析负责精确的结构信息（类名、函数签名、依赖关系）
   - LLM 负责语义理解（模块职责、设计意图、架构模式）
   - 精确数据作为 LLM 的上下文约束，大幅减少幻觉

3. **纯本地，代码零泄露**
   - 所有分析、生成、索引均在本地完成
   - 远程 API 模式下，仅发送结构化的 Prompt 摘要，不发送完整源码
   - 本地 LLM 模式下，100% 离线运行

4. **单二进制，零配置启动**
   - Go 编译为单文件可执行程序，无需安装运行时环境
   - 在项目目录直接运行 `codewiki`，自动完成全流程

**M3 核心痛点回应**：
M2 生成的图表和文档存在严重"清单化"问题——仅有节点列表和图形骨架，缺乏对代码逻辑的深层理解。Zread 的体验证明，开发者真正需要的是"学习体验"而非"结构清单"。M3 通过以下方式解决：

1. **主题导向组织**：文档不按文件/模块罗列，而是按"概念/主题"组织——"What it Does"、"Architecture"、"Project Structure"、"Key Concepts"、"Learning Path"
2. **叙事式内容**：每篇文章回答三个问题——"这是什么？为什么这样设计？对你意味着什么？"
3. **设计决策显性化**：专门提取"Key architectural decisions"，解释为什么选择当前架构、带来了什么好处/代价
4. **学习路径引导**：自动生成"Where to Go Next"，按用户目标（快速上手/部署/理解内部）分支引导
5. **来源溯源**：每个段落末尾标注 Sources，建立文档可信度
6. **函数级语义摘要**：基于 AST 控制流分析 + LLM 生成，描述每个函数的核心职责、输入输出语义、边界条件
7. **模块角色推断**：基于依赖图和 PageRank 分析，推断每个模块在系统中的角色（如"网关层"、"核心领域"、"工具库"）
8. **图表语义层**：在 Mermaid DSL 的 `%%` 注释或文档段落中，为每个图表添加"场景说明"
9. **调用链语义描述**：时序图不再只是消息列表，而是附带"触发条件、各参与者的职责分工、可能的异常分支"
10. **Web UI 体验重塑**：对标 Zread 的产品体验质量，从简陋的技术预览升级为精致的开发者阅读工具

### M3.1 Web UI 体验升级详细设计

> 本节的核心理念：**好的代码文档工具，UI 本身也应该是优秀的阅读工具。** Zread 是我们产品体验的对标对象，我们在内容组织哲学之外，还对 Web UI 进行了全面的体验升级。

#### 设计系统

建立了完整的 CSS 自定义属性设计系统，所有视觉元素通过变量统一控制：

```css
--accent-gradient: linear-gradient(135deg, #6366f1, #8b5cf6, #a78bfa);  /* 品牌渐变色 */
--bg / --bg2 / --bg3       /* 三层背景色阶 */
--text / --text2 / --text3 /* 三层文字色阶 */
--border / --border2       /* 边框层级 */
--shadow / --shadow-md / --shadow-lg  /* 阴影层级 */
--radius / --radius-lg     /* 圆角 */
--pre-bg / --pre-text      /* 代码块配色 */
```

支持亮色/暗色双主题，通过 `[data-theme="dark"]` 选择器覆盖变量，`localStorage` 持久化用户偏好。

#### 页面架构

**Serve 模式（两栏布局）**：
- 左侧：固定侧栏导航 (260px)，磨砂玻璃态
- 右侧：内容区（居中 800px），顶部阅读进度条 + 顶栏

**静态 HTML 模式（三栏布局）**：
- 左栏：固定侧栏导航
- 中栏：文章内容区
- 右栏：文件树 + 目录大纲

#### 渲染双路径统一

### 7.3 LLM 使用策略

```
┌─────────────────────────────────────────────────┐
│            LLM 使用策略分层（双模）               │
│                                                  │
│  ┌──────────────┐  高频调用，需低延迟             │
│  │ 轻量任务      │  - 函数/类摘要生成              │
│  │ (本地小模型    │  - 代码块语义分块               │
│  │  或廉价 API)  │  - API 参数说明生成              │
│  └──────────────┘                                  │
│                                                  │
│  ┌──────────────┐  中等频率，需较强理解            │
│  │ 中等任务      │  - 架构描述生成                 │
│  │ (Sonnet/      │  - Mermaid DSL 生成             │
│  │  Qwen-Max)   │  - 时序图场景推断               │
│  └──────────────┘                                  │
│                                                  │
│  ┌──────────────┐  低频调用，需深度推理            │
│  │ 重型任务      │  - 复杂架构模式识别              │
│  │ (Opus/       │  - 设计意图推断                 │
│  │  Gemini)    │  - 全局一致性审查               │
│  └──────────────┘                                  │
│                                                  │
│  本地模式：全部任务路由到本地 Ollama 模型          │
│  核心原则：缓存 + Prompt 去重，避免重复 LLM 调用   │
└─────────────────────────────────────────────────┘
```

**LLM 调用可靠性架构（v0.8 新增）**：

所有 LLM 调用统一采用 **流式优先** 架构：

```
streamComplete(ctx, provider, prompt)
  ├── 1. provider.CompleteStream(ctx, prompt)     ← 流式优先
  │     ├── 成功 → streamCollectWithLiveness()   ← 活性检测（3 分钟无 token 超时）
  │     │     ├── 正常完成 → 返回响应
  │     │     └── 活性超时 → 降级到步骤 2
  │     └── 失败 → 降级到步骤 2
  └── 2. provider.Complete(ctx, prompt)           ← 非流式降级
        └── 返回响应 / 错误
```

**关键设计决策**：
- **StreamClient**：流式 HTTP 客户端不设 `http.Client.Timeout`，所有超时由 context deadline 控制，避免 HTTP 层先于 context 超时切断连接
- **Thinking 模式**：DeepSeek 使用 `thinking: {type: "enabled"}`，OpenAI 使用 `reasoning_effort: "high"`，两种格式同时发送以兼容 API 网关
- **3 级渐进降级**（章节叙事）：完整 prompt 流式(15min) → 精简 prompt 流式(10min) → 极简 prompt 非流式(8min)
- **Prompt 规模控制**：完整 prompt 模块上限 10 个/函数 3 个；精简 prompt 模块上限 10 个；极简 prompt 模块上限 15 个

## 8. 风险评估与应对

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|
| LLM 幻觉导致文档不准确 | 高 | 高 | AST 精确数据注入约束 Prompt；Mermaid 语法校验 + 自动修复；基准评测集持续监控 |
| LLM 调用超时导致 Wiki 价值丧失 | 中 | 高 | ✅ 已修复：流式优先架构 + StreamClient(context-only timeout) + 3级渐进降级 + 活性检测 + thinking 模式超时适配 |
| 本地 LLM 生成质量不足 | 中 | 中 | 推荐经过验证的模型列表；质量下降时引导用户切换 API 模式；建立模型评测榜单 |
| 大规模仓库性能瓶颈 | 中 | 中 | 文件过滤（排除 vendor/node_modules）；并行 AST 解析；缓存解析结果 |
| tree-sitter 语法覆盖不足 | 低 | 中 | 优先支持成熟语法；对不支持的语言提供 LLM-only fallback |
| Mermaid DSL 生成不稳定 | 中 | 中 | 模板约束 + 语法校验 + 自动修复 + 重试；测试固件锁定预期输出 |
| API 调用成本过高 | 中 | 中 | 缓存 + Prompt 去重；轻量模型做高频任务；本地模型做零成本选择 |
| 竞品（Google）生态碾压 | 中 | 高 | 差异化：可视化 + 本地 + 本地 LLM；Google 不支持纯本地运行 |
| TDD 速度拖慢初期进度 | 中 | 低 | M1 前期允许 Mock 驱动加速；测试固件一次性投入，后续持续受益 |

配置文件 `~/.codewiki/config.yaml`：
```yaml
# 文档生成模型（用于生成 Wiki 文档、图表描述）
generation:
  provider: openai
  api_key: ${OPENAI_API_KEY}
  base_url: https://api.openai.com/v1
  model: gpt-4o
  max_retries: 3
  timeout: 60

# RAG 向量模型（用于代码检索与问答）
embedding:
  provider: openai
  api_key: ${OPENAI_API_KEY}
  base_url: https://api.openai.com/v1
  model: text-embedding-3-small
  max_retries: 3
  timeout: 60

# 向后兼容：旧版单块配置仍可用，会自动同时应用到 generation 和 embedding
```

> **配置说明**：
> - `generation`：负责文档生成、图表描述等创意任务，建议使用强模型（如 GPT-4o、Claude 3.5 Sonnet）。
> - `embedding`：负责代码向量化和 RAG 检索，建议使用专用嵌入模型（如 text-embedding-3-small、nomic-embed-text）。
> - 环境变量支持分离覆盖：`CODEWIKI_GEN_API_KEY` / `CODEWIKI_GEN_MODEL` / `CODEWIKI_GEN_BASE_URL` 和 `CODEWIKI_EMB_API_KEY` / `CODEWIKI_EMB_MODEL` / `CODEWIKI_EMB_BASE_URL`，旧变量 `CODEWIKI_API_KEY` / `CODEWIKI_MODEL` / `CODEWIKI_BASE_URL` 仍可作为两者的 fallback。
